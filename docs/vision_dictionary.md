# MindReply Vision Dictionary

README header: Internal category language for rebuilding MindReply as an Executive Nervous System / Decision Infrastructure Layer. This document controls public copy, product naming, and interaction design.

## Category Definition

MindReply is the layer between input and action. It is not a destination, dashboard, chatbot, or productivity app. It reduces decision friction until the next move is obvious.

## Primary User Feeling

"The next move is obvious."

Every visible surface should create that feeling by reducing choice, reducing pressure, and protecting tone before action happens.

## Page Rules

### Homepage

Friction removed: overload at the moment input arrives.
Decision simplified: what matters now and what to do next.
Trust built: privacy-first language, no inflated claims, no internal operating copy.
Moat: defines the category and anchors unique vocabulary.
Category support: presents MindReply as infrastructure between tools, not another tool.

### MRagent

Friction removed: hesitation around a stuck reply, decision, or booking.
Decision simplified: one synthesis plus one recommended action.
Trust built: calm language, no options, no generic assistant framing.
Moat: private decision filter behavior is harder to copy than a generic answer box.
Category support: demonstrates the nervous-system layer in the smallest possible surface.

### Tools

Friction removed: message and task overload.
Decision simplified: process the one stuck item, then stop or continue.
Trust built: visible limits and clear output, not exaggerated transformation claims.
Moat: tools are playbook surfaces, not disconnected utilities.
Category support: each tool must reduce input to action.

### Bookings

Friction removed: uncertainty about when human review is needed.
Decision simplified: schedule only when risk or specialist judgment requires it.
Trust built: honest scope, professional context, no fake marketplace claims.
Moat: professional playbooks plus specialist review form a trust layer.
Category support: escalation path for the decision infrastructure.

### Memberships

Friction removed: repeated re-explaining and daily decision drag.
Decision simplified: choose access based on repeated pressure, not feature comparison.
Trust built: controlled access and clear memory boundaries.
Moat: private decision memory and signed playbooks deepen over time.
Category support: converts from one-off use to embedded daily behavior.

## Design Rules

- Subtract before adding.
- Show only what matters and what to do next.
- Use navy, charcoal, restrained gold, and high whitespace.
- Avoid noisy dashboards and decorative feature grids.
- Avoid nested cards and internal operating labels.
- Keep labels short, concrete, and action-oriented.

## Output Contract

Default output shape for user-facing decision surfaces:

```text
Synthesis: [1-2 lines]
Recommended action: [reply|schedule|resolve|escalate]. [one next move]
```

No alternatives. No feature explanation. No generic assistant language.
